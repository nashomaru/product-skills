# 開発物をテスト・レビューする

実装が意図した価値と契約を満たし、安全にリリースできるかを証拠に基づいて判定する。

## 原則

- 変更者の説明ではなく、要件、設計、コード、実行結果を根拠にする。
- 差分だけでなく、呼び出し元、データ、権限、失敗経路への影響を確認する。
- テスト成功を正しさの証明とみなさず、テストの妥当性と不足も見る。
- 実行した検証、未実施の検証、推測を区別する。
- 所見を重要度順に示し、場所、再現条件、影響を具体化する。
- レビュー依頼だけの場合は修正しない。修正を依頼された場合にのみ変更する。
- 問題がなくても残余リスクと検証範囲を示す。

## ワークフロー

### 1. 基準と範囲を確定する

要件、設計、タスク、受入基準、実装報告、差分、リポジトリ指示を読む。基準資料がなければ期待動作を確認するか、コード品質レビューとして範囲を限定する。

変更ファイル、関連実行経路、公開契約、データ変更、依存関係、既存テストを特定する。

### 2. リスクベースのテスト計画を作る

次から変更に関係する失敗モードを選ぶ。

- 正常系、境界値、不正入力、権限不足
- 重複、競合、再試行、タイムアウト、部分障害
- データ移行、後方互換、ロールバック
- 性能、負荷、可用性
- セキュリティ、プライバシー、監査
- UIのアクセシビリティ、端末・ブラウザ、状態遷移
- ログ、メトリクス、アラート

変更と無関係な網羅テストを機械的に実行せず、リスクと検証の対応を説明する。

### 3. 静的レビューを行う

要件の未実装、ロジック欠陥、境界・エラー処理、認証・認可、入力検証、情報漏えい、並行性、冪等性、整合性、互換性、移行安全性、性能劣化、テスト不足、不要な複雑さを確認する。スタイルだけの指摘は欠陥や保守リスクにつながる場合に限定する。

### 4. 動的検証を行う

1. 静的解析、型検査、変更箇所のテスト
2. 関連モジュールの統合・回帰テスト
3. 必要なビルド、E2E、性能・セキュリティ検証
4. 受入基準に対応する手動確認

失敗は今回の回帰、既存障害、環境障害に証拠で分類する。分類できなければ未確定とする。

### 5. 所見を評価する

- `P0 Critical`: データ損失、重大な侵害、全体停止。リリース不可。
- `P1 High`: 主要機能の誤動作、重大な回帰・セキュリティ問題。原則リリース不可。
- `P2 Medium`: 限定条件の不具合、現実的な運用・保守リスク。修正または明示的受容が必要。
- `P3 Low`: 影響が小さい改善。通常はリリースを妨げない。

各所見に、問題、場所、発生条件、影響、根拠、推奨修正、要件IDを記載する。

### 6. レポートを作成する

[テスト・レビューレポート](references/test-review-report.md) を使う。所見を先頭に重要度順で示し、その後にテスト結果、要件カバレッジ、判定、残余リスクを記載する。
