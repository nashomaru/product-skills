# 開発タスクを実装する

指定タスクの受入基準を満たす最小で保守可能な変更を実装し、検証可能な状態で引き渡す。

## 原則

- タスク、要件、設計を契約として扱い、矛盾は実装前に明示する。
- 既存コード、規約、テスト、依存関係を調査し、既存パターンを再利用する。
- ユーザーの未コミット変更と無関係なファイルを保持する。
- スコープ外の改善を混ぜない。必要な追加変更は理由と影響を示す。
- 境界で入力を検証し、エラー、権限、並行実行、再試行を設計どおり扱う。
- 受入基準とリスクからテストを同時に実装する。
- 検証していない状態を完了と報告しない。

## ワークフロー

### 1. タスクを確定する

対象タスク、要件ID、設計参照、依存、受入基準、対象外を読む。タスク未指定なら、依存が解消した最優先タスクを提案し、複数候補で結果が変わるときは確認する。

必須依存が未完了、資料が矛盾、重大なプロダクト判断が未決、データ破壊や外部状態変更に新しい権限が必要な場合は着手前に止める。

### 2. 変更箇所を調査する

- 作業ツリーと既存変更を確認する。
- 実行経路、公開契約、データモデル、関連テストを追跡する。
- リポジトリ固有の指示と検証コマンドを確認する。
- 変更するファイルと変更しない境界を定める。

### 3. 実装する

既存の抽象化と命名に合わせ、受入基準を満たす最小の差分を作る。必要に応じて型、入力検証、エラー処理、認可、後方互換、移行、観測性、テスト、文書を含める。

設計から逸脱する必要が判明したら、黙って変更せず、理由、代替、影響を記録して承認を求める。

### 4. 検証する

速く限定的な検証から、リスクに応じて広げる。

1. 静的検査・型検査
2. 追加・更新したテスト
3. 関連モジュールの回帰テスト
4. 必要なビルド、統合、E2E、性能・セキュリティ確認

失敗を隠すためにテストを弱めない。環境理由で実行できない検証は、理由と再現コマンドを残す。

### 5. セルフレビューする

- 差分がタスク範囲内か。
- 受入基準と例外系をテストしているか。
- セキュリティ、整合性、互換性、運用への回帰がないか。
- 一時コード、デバッグ出力、不要な依存、重複がないか。
- 要件、設計、タスクと実装の対応を説明できるか。

### 6. 結果を報告する

[実装報告テンプレート](references/implementation-report.md) を使う。外部Issueの状態を更新するのは、ユーザーが依頼した場合だけにする。
