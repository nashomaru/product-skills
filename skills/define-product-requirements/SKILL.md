# プロダクト要件を定義する

ユーザーと段階的に合意を形成し、実装・設計・テストの判断に使える要件定義書を作成する。

## 基本原則

- 提供済みの会話、文書、コードを先に確認し、既知事項を再質問しない。
- 事実、ユーザーが決めた事項、仮定、提案、未決事項を混同しない。
- 一度に質問する論点を1〜3個に絞る。選択肢を示す場合は差分と影響を添える。
- ソリューションより先に、解決する問題、対象ユーザー、成功状態を確定する。
- 「使いやすい」「高速」などの曖昧語を、観測可能な条件または数値へ変換する。
- スコープ外を明記する。追加要望は既存スコープとのトレードオフを示す。
- 情報不足を推測で埋めない。進行に必要な仮定は明示し、確認待ちにする。
- 技術実装を先に固定しない。ただし既存システムや規制による制約は要件として扱う。

## ワークフロー

### 1. 現状を把握する

既存情報から次を抽出し、「確認済み」「仮定」「未確認」に分類する。

- 背景と解決したい問題
- 対象ユーザーと関係者
- 期待する成果と期限
- 現在の代替手段
- 制約、依存関係、既存資産

情報がほぼない場合は、最初に「誰の、どの問題を、なぜ今解くのか」を聞く。

### 2. 問題と成功を確定する

次の順序で質問し、回答ごとに短く理解を要約する。

1. ユーザー、利用状況、現在の困りごと
2. 問題の頻度・深刻度と既存の回避策
3. 事業・ユーザー成果、成功指標、失敗条件
4. 意思決定者、利用者、運用者などの関係者

機能の希望しか示されていない場合は、その機能が変える行動または成果を確認する。

### 3. スコープと体験を定義する

- 主要ユースケースを、開始条件、基本フロー、代替・例外フロー、終了状態に分解する。
- MVPに必須なものを `Must`、価値はあるが延期可能なものを `Should` / `Could` とする。
- 対象外と将来候補を分ける。
- ロールごとの権限、データの所有者、通知、失敗時の復旧を確認する。
- 重要な業務用語と状態遷移を定義する。

優先順位が競合したら、対象ユーザーへの価値、リスク低減、依存関係、工数の順に比較材料を示し、決定をユーザーに委ねる。

### 4. 要件を検証可能にする

各機能要件に一意なIDを付け、次を記述する。

- 要件: システムが満たす振る舞い
- 根拠: 対応する問題または成果
- 優先度: Must / Should / Could / Won't
- 受入基準: Given / When / Then、または同等に判定可能な条件
- 依存関係・例外: 該当する場合のみ

非機能要件は必要なカテゴリだけ扱い、測定方法を付ける。

- 性能、可用性、拡張性
- セキュリティ、プライバシー、監査、法令
- アクセシビリティ、対応端末・ブラウザ、ローカライズ
- データ保持、移行、バックアップ、復旧
- 監視、運用、サポート

### 5. 整合性を確認する

成果物を出す前に次を確認する。

- 各Must要件が、目的または主要ユースケースへ追跡できる。
- 各要件が単一で、矛盾せず、実現可能で、テスト可能である。
- 正常系だけでなく、入力不備、権限不足、競合、通信失敗、取消・再試行を扱う。
- 成功指標に計測方法、基準値、目標値、評価期間がある。
- 未決事項に担当者または決定期限がある。未定ならその旨を明記する。
- MVPの範囲と対象外が明確である。

重大な未決事項が残る場合は「ドラフト」として提示し、確定版と呼ばない。

### 6. 要件定義書を作成する

[要件定義書テンプレート](references/requirements-template.md) を読み、該当しない節は削除せず `該当なし（理由）` と記載する。出力先を指定されていなければ、チャット上にドラフトを提示して保存先を確認する。既存文書を更新する場合は、その構成と用語を優先する。

## 対話中の出力

各ラウンドでは次の短い形式を使う。

```markdown
理解したこと: ...
未決事項: ...
次の質問:
1. ...
```

会話が長くなったら、随時「決定ログ」を示す。ユーザーが「仮定して進めて」と指示した場合のみ、仮定を列挙してドラフト作成へ進む。
