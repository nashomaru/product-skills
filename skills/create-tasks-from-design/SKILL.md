# 設計から開発タスクを作成する

設計を、担当者が追加の設計判断を最小限にして着手・完了判定できるタスク列へ変換する。

## 原則

- 要件IDと設計要素を各タスクへ引き継ぐ。
- タスクを成果単位で分ける。「バックエンド対応」など責務名だけのタスクを作らない。
- 可能な限り小さな垂直スライスにし、単独で検証可能にする。
- 依存関係を明示し、循環を作らない。単なる作業順を依存と混同しない。
- 実装、テスト、移行、観測性、文書を残作業にしない。
- 不確実性が大きい場合は、期限と成果物を定めた調査タスクを先行させる。
- 見積値を根拠なく生成しない。求められた場合は相対規模と不確実性を示す。

## ワークフロー

### 1. 入力を検査する

設計書、要件定義、既存Issue規約、リポジトリ構成を読む。重要な契約が未決、受入基準がない、要件を追跡できない場合は、影響を示して差し戻す。

### 2. 成果と変更面を抽出する

ユーザーに届く振る舞いと、その実現に必要なデータ、API、UI、セキュリティ、運用、移行、テストを列挙する。設計書の章をそのままタスクにしない。

### 3. タスクへ分解する

各タスクをおおむね1回のレビュー可能な変更で完了できる粒度にし、次を含める。

- 目的と利用可能になる成果
- 要件IDと設計参照
- 前提、依存、対象範囲、対象外
- 変更するコンポーネントや契約
- 検証可能な受入基準
- 必要なテスト、移行、観測性、文書

共同変更しないと一貫性を保てないものは、無理に分離しない。

### 4. 順序と依存を決める

未知を減らす調査、スキーマ・契約・共有基盤、機能スライス、統合・移行・展開の順を基準にする。並列実行可能なタスクを明示する。公開契約やデータ変更には互換期間とロールバックを含める。

### 5. カバレッジを検査する

- すべてのMust要件と設計要素がタスクに対応する。
- 例外系と非機能要件を検証する作業がある。
- セキュリティ、アクセシビリティ、監視、文書、リリースが漏れていない。
- 各タスクが後続に必要な契約を提供する。
- 重複と循環依存がない。

### 6. タスク一覧を作成する

[開発タスクテンプレート](references/task-template.md) を読み、既存のIssueテンプレートがあれば適合させる。外部ツールへの登録は、ユーザーが依頼した場合だけ行う。
