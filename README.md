# Product Development Skills

プロダクトの要件定義から設計、タスク分解、実装、独立QAまでを一貫して進めるためのCodexスキル集です。

開発タスクが準備できた後は、PMスキルが実行可能なタスクを最大3件選び、開発と独立QAを並列管理します。

## 全体フロー

```text
プロダクト構想
    │
    ▼
要件定義
    │
    ▼
技術設計
    │
    ▼
開発タスク作成
    │
    ▼
PMによる3レーン選定
    │
    ├─ Lane A: 開発 ─┐
    ├─ Lane B: 開発 ─┼─ 並列実行
    └─ Lane C: 開発 ─┘
                      │
                      ▼
    ├─ Lane A: 独立QA ─┐
    ├─ Lane B: 独立QA ─┼─ 担当を交換して並列実行
    └─ Lane C: 独立QA ─┘
                      │
                      ▼
              修正 → 再QA
                      │
                      ▼
              統合テスト・完了判定
```

要件定義から開始する必要はありません。すでに設計や開発タスクがある場合は、該当する段階から開始できます。

| [`create-tasks-from-design`](skills/create-tasks-from-design/SKILL.md) | 設計を依存関係付きの実行可能なタスクへ分解する | 開発タスク一覧 |

## 最短の使用方法

開発タスクまで作成済みの場合は、次のようにPMスキルを起動します。

```text
$manage-parallel-delivery を使って、
docs/tasks.md のReadyな開発タスクを最大3件ずつ処理してください。
開発、独立QA、修正、再検証まで自動で進めてください。
```

タスクファイルのパスは実際の保存先へ変更してください。タスクがチャットやIssueにある場合は、対象を明示します。

```text
$manage-parallel-delivery を使って、TASK-101、TASK-102、TASK-103を処理してください。
要件は docs/requirements.md、設計は docs/design.md を参照してください。
```

PMはタスクを再作成せず、次の処理から開始します。

1. タスクと依存関係を確認する。
2. `Ready` なタスクを最大3件選ぶ。
3. 開発ワーカーを並列起動する。
4. 実装結果を確認する。
5. 開発担当とは別のQA担当を割り当てる。
6. QAを並列実行する。
7. 必須指摘を開発担当へ差し戻す。
8. 修正後に同じQA担当が再検証する。
9. 統合後のテストを実行し、完了を判定する。

## 開発タスクに必要な情報

自動実行の精度を保つため、各タスクには次の情報を含めてください。

```markdown
## TASK-101: ログイン試行回数を制限する

### 目的・成果

短時間の連続したログイン失敗を制限し、不正アクセスを抑止する。

### 要件・設計参照

- 要件: FR-012、NFR-004
- 設計: ADR-007、設計書 6.2

### 依存関係

- TASK-099が完了していること

### 対象範囲

- ログインAPIの試行回数制御
- 制限時のエラーレスポンス
- 単体テストと統合テスト

### 対象外

- 管理画面からの制限解除

### 受入基準

- Given: 同じアカウントで5回連続して認証に失敗している
- When: 6回目のログインを試行する
- Then: 認証処理を行わず、規定のエラーを返す

### 検証

- 境界値となる5回目と6回目をテストする
- 制限期間終了後に再試行できることを確認する
- 異なるアカウントへ影響しないことを確認する
```

タスクの詳細な形式は[開発タスクテンプレート](skills/create-tasks-from-design/references/task-template.md)を参照してください。

### タスク状態の判定

PMはタスクを次の状態に分類します。

| 状態 | 意味 | PMの動作 |
|---|---|---|
| `Ready` | 依存が解決し、受入基準と検証方法が明確 | 開発候補として選択する |
| `Blocked` | 依存、権限、環境、意思決定を待っている | ブロッカーを報告して待機する |
| `Needs refinement` | 実装結果を左右する情報が不足している | タスクの明確化を要求する |
| `Done` | 実装と独立QAが完了している | 再実行しない |

## 3レーン並列実行

PMは同時に最大3件を扱います。3件に満たない場合は、存在するタスクだけを実行します。

### 並列化できる例

```text
Lane A: ユーザー設定API
Lane B: 通知メールテンプレート
Lane C: 管理画面の検索UI
```

変更対象が分離され、直接依存がなければ並列実行できます。

### 並列化しない例

```text
TASK-201: DBスキーマを追加
TASK-202: 追加されたスキーマを使用するAPIを実装
```

TASK-202がTASK-201に依存する場合、TASK-201を先に完了させます。

次の場合も無理に並列化しません。

- 同じファイルを相反する形で変更する。
- 同じAPI契約を別々に変更する。
- マイグレーションの実行順が決まっている。
- テストが同じDB、ポート、生成物を共有し、隔離できない。

競合の可能性がある実装は別ブランチと別worktreeで隔離します。テスト資源を隔離できない場合、そのテストだけ順次実行します。

## 開発ウェーブ

各開発ワーカーは[`implement-development-task`](skills/implement-development-task/SKILL.md)を使用し、1件のタスクだけを担当します。

開発ワーカーは次を行います。

- タスク、要件、設計、既存コードを確認する。
- 指定された編集境界内で実装する。
- 受入基準に対応するテストを追加・更新する。
- 静的検査、対象テスト、必要な回帰テストを実行する。
- 変更内容、検証結果、未解決事項を報告する。

PMは次の状態ではQAへ進めません。

- 必須テストが失敗している。
- 受入基準に対応する実装またはテストがない。
- タスク範囲外の変更が混在している。
- 他レーンとの競合がある。
- 未承認の設計逸脱がある。

## 独立QAウェーブ

開発完了後、[`test-and-review-implementation`](skills/test-and-review-implementation/SKILL.md)を使用して独立QAを行います。

3レーンの場合は担当を次のように交換します。

```text
開発担当 A → Lane BをQA
開発担当 B → Lane CをQA
開発担当 C → Lane AをQA
```

2レーンの場合は相互に交換します。開発者本人によるセルフテストだけで最終完了にはしません。

QAは次を確認します。

- 要件と受入基準を満たしているか。
- 正常系、境界値、異常系が検証されているか。
- 権限、データ整合性、並行実行、再試行に問題がないか。
- セキュリティ、性能、アクセシビリティへの回帰がないか。
- テスト自体が適切な失敗を検出できるか。

### QA所見の扱い

| 重要度 | 例 | 完了条件 |
|---|---|---|
| `P0 Critical` | データ損失、重大な侵害、全体停止 | 必ず修正する |
| `P1 High` | 主要機能の誤動作、重大な回帰 | 原則として必ず修正する |
| `P2 Medium` | 限定条件の不具合、現実的な運用リスク | 修正または明示的な受容が必要 |
| `P3 Low` | 影響が小さい改善 | 後続タスク化できる |

修正後は、元の指摘を出したQA担当が再検証します。

## 完了条件

タスクは次の条件をすべて満たした場合だけ完了になります。

- 受入基準を満たす実装がある。
- 必須の静的検査、テスト、ビルドが成功している。
- 独立QAがリリース可能と判定している。
- 未解決のP0/P1がない。
- 未受容のP2がない。
- 未実施の検証と残余リスクが記録されている。
- 統合後に共有契約と変更経路を再検証している。

進捗記録には[並列デリバリーボード](skills/manage-parallel-delivery/references/delivery-board.md)を使用します。

## PMが自動判断せず停止する条件

次の場合はユーザーの判断や権限が必要です。

- 要件やプロダクトのスコープを変更する必要がある。
- 技術設計の重要な方針を変更する必要がある。
- データ破壊や後方互換性の喪失が発生する。
- 外部サービスや本番環境を変更する必要がある。
- 新しい認証情報や権限が必要である。
- 依存タスクや必要な環境が利用できない。
- タスク間の優先順位を技術情報だけでは決定できない。

push、Pull Request作成、デプロイ、外部Issue更新は、ユーザーが明示的に依頼した場合だけ実行します。

## 段階別の使用例

### 構想から開始する

```text
$define-product-requirements を使って、予約管理機能の要件をヒアリングしてください。
```

要件確定後:

```text
$design-from-requirements を使って、docs/requirements.mdから技術設計を作成してください。
```

設計確定後:

```text
$create-tasks-from-design を使って、docs/design.mdから開発タスクを作成してください。
```

タスク作成後:

```text
$manage-parallel-delivery を使って、Readyなタスクを最大3件ずつ実装し、独立QAまで進めてください。
```

### 1件だけ実装する

```text
$implement-development-task を使って、TASK-101を実装してください。
要件はdocs/requirements.md、設計はdocs/design.mdを参照してください。
```

### 実装済み変更だけレビューする

```text
$test-and-review-implementation を使って、TASK-101の実装をレビューしてください。
修正は行わず、所見とテスト結果を報告してください。
```

### すべてのReadyタスクを継続処理する

```text
$manage-parallel-delivery を使って、Readyなタスクを優先度順に処理してください。
最大3レーンで開発と独立QAを繰り返し、BlockedまたはNeeds refinement以外がなくなるまで進めてください。
```

## ディレクトリ構成

```text
.
├── README.md
├── agents/
│   ├── delivery-pm.toml
│   └── requirements-analyst.toml
└── skills/
    ├── define-product-requirements/
    ├── design-from-requirements/
    ├── create-tasks-from-design/
    ├── implement-development-task/
    ├── test-and-review-implementation/
    └── manage-parallel-delivery/
```

各スキルのディレクトリには次を配置しています。

- `SKILL.md`: 適用条件、原則、実行手順
- `agents/openai.yaml`: UI表示名と既定プロンプト
- `references/`: 成果物や報告書のテンプレート

`agents/delivery-pm.toml` は、並列デリバリー専用のPMエージェント定義です。

## 運用上の推奨事項

- 要件ID、設計判断ID、タスクIDを維持し、成果物間を追跡可能にする。
- タスクは1回のレビューで扱える大きさにする。
- タスク本文に対象外を明記してスコープ拡大を防ぐ。
- テストを後続タスクに分離せず、各実装タスクの完了条件へ含める。
- 共有契約やマイグレーションを変更するタスクは、統合順を先に決める。
- QAの未実施項目をテスト成功と混同しない。
- 自動処理を長時間継続する場合も、各ウェーブの完了時に証拠と残余リスクを記録する。
